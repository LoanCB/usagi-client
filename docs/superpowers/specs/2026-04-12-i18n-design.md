# i18n — Design spec

Date: 2026-04-12

## Goal

Add French/English internationalisation to the Usagi Tauri desktop app using `react-i18next`. The UI is currently in French with hardcoded strings; this spec describes migrating every string to translation keys and adding a language toggle.

## Supported languages

| Code | Language | Notes |
|------|----------|-------|
| `fr` | French | Current UI language, becomes first supported locale |
| `en` | English | Second locale; also used as i18next fallback |

## Architecture

### Packages

```text
i18next
react-i18next
i18next-browser-languagedetector
```

No extra packages. TypeScript locale files are used instead of JSON to get type-checking without an additional plugin.

### File structure

```text
src/i18n/
  index.ts            ← i18next init (detector, fallback, resources)
  types.d.ts          ← i18next type augmentation for typed t()
  locales/
    en.ts             ← English strings (reference shape)
    fr.ts             ← French strings (must match en.ts shape)
```

### Initialisation

`src/i18n/index.ts` is imported as a **side-effect** at the top of `src/main.tsx`, before React renders. This ensures translations are synchronously available on first render (no loading flicker).

```ts
// main.tsx
import './i18n'
```

### Detection & persistence

`i18next-browser-languagedetector` runs in order:
1. `localStorage` — persisted user preference
2. `navigator` — OS/browser locale (works in Tauri via Chromium's `navigator.language`)
3. Fallback to `en`

Supported languages: `['fr', 'en']`. The detector strips region codes (`fr-FR` → `fr`).

### Namespace

Single namespace: `translation` (i18next default). ~60 strings across 10 components — no benefit to splitting.

## Type safety

`src/i18n/types.d.ts` augments the `i18next` module so that `t('task.new')` is checked at compile-time:

```ts
import type en from './locales/en'

declare module 'i18next' {
  interface CustomTypeOptions {
    defaultNS: 'translation'
    resources: { translation: typeof en }
  }
}
```

Any call to `t()` with an unknown key produces a TypeScript error. The French locale (`fr.ts`) must satisfy `typeof en` — TypeScript enforces structural completeness.

## Language toggle UI

A `LanguageToggle` component is added to the bottom of `Sidebar`, outside the `ScrollArea`:

```
┌─────────────────┐
│ [scroll area]   │
├─────────────────┤
│    FR  |  EN    │  ← fixed bottom strip
└─────────────────┘
```

- Two buttons styled like tabs; active locale is highlighted
- Calls `i18n.changeLanguage(lang)` on click
- Collapsed sidebar: shows only the active locale code (e.g. `FR`)
- No separate settings page needed

## Constraints

- No lazy-loading of translation files (synchronous import — ~2kb each, Tauri desktop, no network cost)
- No pluralisation rules needed (French/English share the same plural forms for the strings in scope)
- `date-fns` locale is **not** in scope for this spec (dates are already formatted with `formatDate()` helper — a follow-up can pass the locale there)
